# AgeCalc SEO·AdSense·라이프 플랫폼 설계

작성 기준일: 2026-06-19  
대상 서비스: https://agecalc.cloud  
전략 방향: **AdSense 승인 우선 혼합형 — 기존 품질 개선 후 검증된 클러스터만 확장**

> 이 문서의 검색 수요는 Search Console·Google Ads Keyword Planner 원본 데이터가 없는 상태에서 공개 검색 의도와 일반적인 한국어 키워드 수요를 기준으로 추정한 범위다. 실제 개발 착수 전 Search Console과 Keyword Planner로 다시 보정해야 한다.

## 1. 현재 사이트 진단

### 1.1 확인된 운영 상태

저장소와 Flask 앱을 기준으로 확인한 현재 상태는 다음과 같다.

| 항목 | 현재 상태 | 평가 |
|---|---|---|
| 공개 사이트맵 | 50개 URL | 소규모 도구 사이트를 넘어선 수준이나 페이지 간 가치 편차가 큼 |
| 정적 가이드 | 20개 | 주제 연관성은 좋지만 상당수가 짧고 동일한 형식을 반복 |

### 1.2 현재 사이트의 강점

1. **주제가 명확하다.** 만나이에서 출발해 학년, 출생연도, 월령, 기념일, 가족 관계로 확장할 수 있다.
2. **직접 작동하는 도구가 있다.** 텍스트만 있는 정보 사이트보다 사용자의 문제를 즉시 해결한다.
3. **한국 생활 제도와 연결하기 쉽다.** 입학, 학년, 연금, 정년, 건강검진 등 반복 검색되는 영역과 자연스럽게 연결된다.
4. **서버 렌더링 기반이다.** Flask/Jinja 페이지는 검색 엔진이 핵심 본문과 링크를 읽기 쉽다.
5. **색인 통제가 이미 존재한다.** 블로그와 미니게임을 무조건 공개하지 않는 구조는 향후 품질 게이트로 발전시키기 좋다.
6. **계산 기준과 운영 원칙 페이지가 있다.** E-E-A-T를 강화할 기반이 마련되어 있다.

### 1.3 현재 사이트의 핵심 약점

#### A. 페이지 수에 비해 독립적인 정보 가치가 약하다

계산기와 표는 다르지만, 검색 엔진 관점에서 보면 상당수 페이지가 다음 형식을 공유한다.

- 연도 또는 날짜 입력
- 결과 숫자 표시
- 짧은 계산 공식
- 비슷한 예시
- 몇 개의 관련 링크

이 구조가 반복되면 전체 사이트가 “하나의 계산 기능을 여러 키워드로 나눈 사이트”처럼 보일 수 있다. 페이지마다 고유한 사용자 과제, 해설, 공식 근거, 다음 행동이 있어야 한다.

#### B. 가이드가 계산기 설명서 수준에 머문다

20개 가이드는 주제 적합성이 높지만 다수의 본문이 약 600~800자 수준의 짧은 문단 묶음이다. 검색 결과에 이미 존재하는 일반 상식을 재정리한 수준으로 평가될 가능성이 있다. 특히 동일한 5개 섹션과 2개 FAQ 형식이 반복되면 기계적으로 생산된 인상을 줄 수 있다.

#### C. 사이트의 정체성이 아직 “라이프 플랫폼”보다 “계산기 모음”에 가깝다

홈페이지 문구는 Life Calendar를 지향하지만 실제 정보 구조는 계산기, 표, 안내로만 나뉜다. 사용자가 나이를 계산한 뒤 무엇을 준비하고 확인할지 이어지는 생애 사건 정보가 부족하다.

#### D. 상업 요소가 콘텐츠보다 앞설 수 있다

쿠팡 캐러셀이 공통 헤더 하단에서 호출되는 구조다. 활성화 시 첫 화면에서 사이트 자체 콘텐츠보다 긴 광고 레일과 상품 배너가 먼저 인지될 수 있다. AdSense 재심사 전에는 “사용자 문제 해결이 주목적”이라는 인상을 명확히 해야 한다.

#### E. YMYL 확장에 필요한 편집 체계가 부족하다

연금, 정년, 건강검진, 임신·출산은 검색 수요와 광고 가치가 높지만 법률·재정·의료 판단에 영향을 줄 수 있다. 현재의 일반 운영 원칙만으로는 최신성 관리, 검수자, 공식 출처, 변경 이력 관리가 부족하다.

## 2. AdSense 승인 가능성 평가

### 2.1 현재 승인 가능성

**평가: 낮음~보통 이하**

기술적 접근성이나 필수 정책 페이지는 큰 문제가 아니다. 반복 거절의 더 유력한 원인은 사이트 전체에서 확인되는 독립적인 편집 가치와 콘텐츠 깊이다. AdSense 승인은 특정 글자 수나 페이지 수로 보장되지 않는다. 현재 50개 URL이 있어도 심사자가 “비슷한 계산 결과와 짧은 설명이 반복되는 도구 모음”으로 판단하면 Low Value Content 사유가 유지될 수 있다.

### 2.2 주요 거절 가능 원인

| 위험 요인 | 현재 위험도 | 판단 근거 | 대응 |
|---|---:|---|---|
| 결과 숫자 외 독립 가치 부족 | 높음 | 계산하지 않으면 읽을 이유가 약한 페이지가 존재 | 해설·사례·공식 기준·다음 행동 추가 |
| 유사 템플릿 반복 | 높음 | 표와 가이드가 동일한 문장 구조를 반복 | 페이지별 고유 섹션·데이터·시각화 적용 |
| 짧은 가이드 | 높음 | 다수가 검색 의도를 완결하기에 짧음 | 대표 가이드를 우선 1,500~3,000자 수준으로 재편 |
| 광고·제휴 우선 인상 | 높음 | 공통 헤더 직후 쿠팡 레일 가능 | 승인 전 전역 제휴 제거 또는 본문 하단 제한 |
| 작성 책임과 검수 정보 부족 | 중간 | 편집팀 표시는 있으나 페이지별 작성·검수·수정일 부족 | author/reviewer/dateModified/source 명시 |
| 사이트 주제 희석 | 낮음~중간 | 미니게임은 별도 존재하지만 noindex 처리됨 | 현 상태 유지, 주 탐색에서 제거 |
| 자동 생성 콘텐츠 위험 | 중간 | RSS·OpenAI 블로그 파이프라인 존재 | 수동 검수·고유 경험·출처·색인 게이트 유지 |
| 검색 의도 중복 | 중간~높음 | 만나이·연나이·나이표·비교표가 경쟁 가능 | 대표 URL과 보조 URL 역할 분리 |

### 2.3 개선 후 승인 가능성

다음 조건을 모두 만족한 뒤 재심사하면 **보통~높음** 수준까지 개선할 수 있다.

- 전역 쿠팡 제휴 노출을 중단하고 핵심 콘텐츠가 첫 화면을 차지한다.
- 핵심 20~30개 URL이 계산 결과 없이도 유용한 독립 본문을 갖는다.
- 유사하거나 얇은 URL은 통합, noindex 또는 sitemap 제외한다.
- 주요 페이지에 작성자, 검수자, 기준일, 공식 출처, 수정 이력을 표시한다.
- 허브 8개와 문맥형 내부링크를 구축한다.
- 자동 블로그는 충분한 품질과 수동 검수 기록이 생길 때까지 noindex를 유지한다.
- 재심사 직전 최소 2~4주 동안 대규모 URL 변경 없이 안정적으로 운영한다.

### 2.4 재심사 전 체크리스트

- [ ] 색인 대상 URL마다 고유 검색 의도가 한 문장으로 정의되어 있다.
- [ ] 각 핵심 페이지에 최소 3개의 고유 정보 요소가 있다.
- [ ] 공식 기준이 필요한 페이지는 원문 출처와 확인일이 표시된다.
- [ ] 전역 쿠팡 레일과 과도한 제휴 배너가 제거되어 있다.
- [ ] 빈 광고 슬롯이나 레이아웃을 밀어내는 광고 영역이 없다.
- [ ] title·H1·본문이 같은 검색 의도를 설명한다.
- [ ] 작성자·검수자·최종 수정일이 페이지에 보인다.
- [ ] sitemap에는 index 가능한 200 응답 canonical URL만 들어 있다.
- [ ] noindex 페이지는 sitemap과 주 탐색에서 제외되어 있다.
- [ ] 모바일 첫 화면에서 사이트 기능과 설명이 광고보다 먼저 보인다.
- [ ] Search Console의 중복·soft 404·크롤링 오류가 정리되어 있다.
- [ ] 정책 페이지와 문의 경로가 모든 공개 페이지에서 접근 가능하다.

## 3. 문제점

### 3.1 콘텐츠 문제

1. 계산 공식과 짧은 FAQ가 여러 URL에서 반복된다.
2. 출생연도·학년·띠 데이터가 표로만 존재해 해석과 행동 정보가 약하다.
3. 실제 사용자 사례, 비교 사례, 체크리스트, 다운로드 자료가 부족하다.
4. “누가, 언제, 어떤 근거로 검수했는가”가 페이지별로 명확하지 않다.
5. 최신 제도 변화에 대한 업데이트 주기와 변경 이력 체계가 없다.
6. 계산 결과 페이지에 결과에 맞는 개인화된 관련 콘텐츠가 충분히 연결되지 않는다.
7. 공유할 수 있는 결과 카드와 저장 가능한 산출물이 제한적이다.

### 3.2 SEO 문제

1. 사이트 구조가 계산기/표·비교/안내로만 나뉘어 사용자 생애 과제를 반영하지 못한다.
2. 유사 검색 의도 URL 사이의 대표 페이지가 명확하지 않다.
3. 전역 메뉴 링크는 많지만 본문 내 의미 있는 내부링크가 부족하다.
4. 일부 페이지에만 구조화 데이터가 있고 Breadcrumb 계층이 없다.
5. 날짜가 바뀌는 페이지의 `lastmod`, 기준연도, 수정 정책이 일관되지 않다.
6. 쿼리 파라미터로 선택한 결과를 별도 URL로 색인할지에 대한 정책이 없다.
7. 카테고리 허브가 없어 개별 페이지의 문맥과 주제 권위가 약하다.
8. 모든 URL을 한 사이트맵에 넣어 클러스터별 색인 성과를 분리하기 어렵다.

### 3.3 내부링크 문제

- 헤더가 대부분의 페이지를 연결하지만, 검색 엔진에는 사이트의 중요도 차이가 약하게 전달된다.
- 관련 링크가 페이지 하단에만 있거나 동일한 링크 묶음을 반복한다.
- 결과에 따라 달라지는 링크가 거의 없다. 예를 들어 1964년생 결과에는 연금·정년·건강검진 콘텐츠가 연결되어야 한다.
- 가이드에서 계산기로는 연결되지만 계산기 결과에서 심층 가이드와 라이프 이벤트로 이어지는 흐름이 약하다.
- 상위 허브 → 대표 계산기 → 상세 가이드 → 인접 도구 → 상위 허브의 순환 구조가 없다.

### 3.4 Search Console 관점 문제

- 실제 색인 품질보다 sitemap URL 수를 늘리는 방향으로 흐를 위험이 있다.
- 연도별 URL을 대량 생성하면 `발견됨 - 현재 색인 생성 안 됨`이 급증할 수 있다.
- 템플릿 유사도가 높으면 `크롤링됨 - 현재 색인 생성 안 됨` 또는 중복 페이지로 분류될 수 있다.
- 검색 의도가 겹치는 페이지가 서로 노출을 나눠 갖는 키워드 카니벌라이제이션 가능성이 있다.
- GSC 성과를 허브별로 비교할 수 있는 sitemap 분리가 없다.

### 5.3 내부링크 원칙

각 핵심 페이지는 다음 5종 링크 중 최소 3종을 가져야 한다.

1. **상위 허브 링크:** Breadcrumb와 본문 도입부에서 주제 위치를 설명한다.
2. **계산 전 링크:** 입력 전에 기준을 이해할 가이드로 연결한다.
3. **결과 기반 링크:** 결과 연령·출생연도·생애 단계에 맞는 다음 페이지를 제안한다.
4. **인접 도구 링크:** 같은 데이터를 다시 입력하지 않고 관련 계산으로 이동한다.
5. **공식 출처 링크:** 제도·의료·재정 정보는 원문 기관으로 연결한다.

권장 흐름:

```text
검색 유입
  → 계산기 또는 가이드
  → 개인 결과와 해설
  → 해당 연령의 라이프 이벤트
  → 체크리스트·관련 계산기
  → 공유 또는 재방문
```

```text
/sitemap.xml

├── /sitemaps/age.xml
├── /sitemaps/family.xml
├── /sitemaps/education.xml
├── /sitemaps/anniversary.xml
```

- sitemap에는 canonical, 200 응답, index 가능 URL만 포함한다.
- `lastmod`는 실제 본문·기준 데이터가 변경된 날만 갱신한다.
- 필터, 정렬, 입력 결과 쿼리 URL은 기본적으로 색인하지 않는다.
- 연도별 URL은 품질 게이트를 통과한 연도만 sitemap에 추가한다.

## 6. 콘텐츠 확장 전략

### 6.1 신규 페이지 품질 게이트

새 URL은 아래 7개 항목 중 필수 5개와 선택 2개를 모두 충족할 때만 index한다.

**필수**

1. 다른 URL과 구분되는 독립 검색 의도
2. 사용자 입력 또는 고유 데이터에 따른 실제 기능
3. 계산 결과를 해석하는 고유 본문
4. 기준일과 출처
5. 관련 다음 행동 또는 체크리스트

**선택**

6. 고유 표·차트·타임라인
7. 공유 카드·인쇄·다운로드

단순히 제목과 숫자만 바꾼 연도별 페이지, AI가 생성한 일반론, 검색 결과를 재요약한 글은 index하지 않는다.

### 6.4 중요 아이디어 발전안

#### 출생연도별 나이 페이지

단순 나이표가 아니라 다음 요소를 조합한 “출생연도 프로필”로 만든다.

- 현재 만나이 범위와 연나이
- 띠와 세대 분류
- 초·중·고·대학 입학 추정 연도
- 환갑·연금·정년 등 주요 예정 시점
- 해당 연도의 사회·문화 사건
- 같은 해 태어난 유명인
- 공식 기준일과 예외

모든 연도를 한꺼번에 index하지 않는다. 우선 Search Console에서 실제 쿼리가 확인되는 10~20개 연도만 출시한다.

#### 세대별 추억 탐험

출생연도를 입력하면 초등학교, 중학교, 스무 살 무렵의 연도와 당시의 대중문화·기술·생활상을 보여준다. 사용자가 기억 항목을 선택해 공유 카드를 만들 수 있게 한다. 데이터 출처와 저작권 상태가 명확한 텍스트·공공 데이터를 사용하며, 이미지 무단 수집은 피한다.

#### 나이별 인생 타임라인

출생일부터 현재까지의 생일, 학교 시기, 법적 성년, 주요 기념 나이를 한 화면에 표시한다. 결과는 개인에게 고유하고 공유 가치가 높다. 제도 항목은 일반 안내로 한정하고 현재 기준 공식 출처를 연결한다.

#### 나와 동갑 연예인

위키·검색 결과를 무단 복제하지 않고 라이선스가 명확한 구조화 데이터 또는 직접 구축한 검수 DB를 사용한다. 국적·직업·생일 기준 필터를 제공하며, 정확한 생년월일과 출처를 표시한다.

#### 연금·정년 계산

출생일만으로 확정적인 금액이나 법적 권리를 판단하지 않는다. 가입 유형, 직업, 고용 규정에 따라 달라질 수 있음을 명확히 표시하고 국민연금공단·국가법령정보센터 등 공식 원문을 연결한다. 기준 개정 시 자동 경고가 발생하도록 출처 점검일을 관리한다.

#### 나이별 건강검진

“검사를 받아야 한다”는 진단 표현보다 “일반적인 국가검진 대상 기준을 확인한다”는 정보 도구로 제한한다. 성별, 출생연도, 가입 유형 등 필요한 조건을 입력받고, 결과마다 국민건강보험공단·질병관리청 원문을 연결한다.

### 6.5 콘텐츠 템플릿

모든 중요 계산기·가이드는 다음 순서를 기본으로 한다.

1. 검색 질문에 대한 2~3문장 직접 답변
2. 입력 또는 핵심 결과
3. 결과의 의미와 계산 공식
4. 실제 사례 2~4개
5. 예외·주의사항
6. 공식 출처와 기준일
7. 관련 체크리스트
8. 결과 기반 내부링크
9. 작성·검수·수정 정보

### 6.6 Search Console 색인 확장 전략

#### 출시 방식

- 한 번에 100개 URL을 공개하지 않고 5~10개 단위로 배치한다.
- 배치마다 2~4주간 크롤링, 색인, 노출, 카니벌라이제이션을 확인한다.
- 신규 URL은 허브와 최소 2개 기존 페이지에서 링크한 뒤 sitemap에 넣는다.
- 8주 동안 노출이 전혀 없고 독립 가치가 약한 URL은 통합 또는 noindex 후보로 검토한다.

#### 상태별 대응

| GSC 상태 | 의미 | 대응 |
|---|---|---|
| 발견됨 - 현재 색인 생성 안 됨 | 우선순위·크롤링 자원 부족 가능 | 내부링크 강화, sitemap 배치 축소, 얇은 URL 제거 |
| 크롤링됨 - 현재 색인 생성 안 됨 | 품질·중복·의도 약함 가능 | 본문 고유성, 대표 의도, 데이터 가치 재검토 |
| 중복, Google이 다른 canonical 선택 | URL 역할 또는 canonical 불명확 | 파라미터·유사 페이지 통합, 내부링크 canonical 일치 |
| soft 404 | 실질 정보가 부족하거나 결과 없음 | 빈 결과 페이지를 404/noindex 또는 유용한 대체 안내로 전환 |
| 적절한 canonical이 있는 대체 페이지 | 정상일 수 있음 | 의도한 대표 URL인지 확인 |

## 7. AI 기능 확장 전략

### 7.1 기본 원칙

- AI는 기존 계산 결과를 대체하지 않고 설명·정리·개인화하는 계층으로 사용한다.
- 날짜와 제도 계산은 결정적 코드와 검증된 데이터로 수행하고, 모델은 그 결과를 자연어로 설명한다.

- 공개 콘텐츠 생성에 AI를 사용할 경우 출처 검증과 사람의 편집 승인을 필수로 한다.

- 피부 나이 분석
- 얼굴 노화·회춘·미래 얼굴 생성

이미지 실험군은 AdSense 승인 전 핵심 제품에 포함하지 않는다. 생체 이미지 처리 고지, 미성년자 정책, 저장·삭제 정책, 오락 목적 표시, 유해 출력 차단을 먼저 설계해야 한다.

### 7.4 AI 기술 설계 원칙

- 텍스트 생성과 도구 호출은 OpenAI Responses API 기반으로 구성한다.
- 날짜 계산, 학년 계산, 연금 기준 판정은 서버 함수가 수행하고 모델에는 검증된 결과만 제공한다.
- JSON Schema 기반 구조화 출력을 사용해 제목, 요약, 주의사항, 링크 후보를 고정된 필드로 받는다.
- 사용자 입력과 생성 결과는 Moderation 및 자체 정책 필터를 거친다.
- 이미지 기능은 이미지 입력·생성 모델을 분리하고 원본 사진의 기본 보관을 하지 않는다.
- 요청 ID, 모델, 프롬프트 버전, 출처 버전, 출력 검수 상태를 기록한다.
- 공개 글 자동 게시를 금지하고 사람의 승인 이후에만 index한다.

## 운영 정책

### YMYL 페이지 필수 항목

연금·정년·건강·임신 페이지는 다음을 모두 포함한다.

- 정보 기준일
- 공식 기관 원문 링크
- 적용 대상과 제외 대상
- 일반적인 계산과 개별 판단의 차이
- 작성자와 검수자
- 최종 수정일과 변경 이력
- “공식 판정·진단·상담을 대체하지 않는다”는 명확한 안내

### 광고와 제휴 원칙

- 첫 화면의 주 목적은 계산 또는 정보 제공이어야 한다.
- 자동 광고가 입력 폼과 결과 사이를 방해하지 않도록 한다.
- 제휴 상품은 페이지 주제와 직접 관련된 경우에만 본문 하단에 배치한다.
- 광고·제휴 링크에는 명확한 표시와 필요한 `sponsored` 속성을 사용한다.
- 승인 전에는 광고 단위 수 최적화보다 콘텐츠 가치와 탐색성을 우선한다.

### 프로그램매틱 SEO 원칙

- 연도·나이·학년 조합을 무제한 URL로 생성하지 않는다.
- 검색 수요, 고유 데이터, 고유 본문, 내부링크가 확보된 URL만 공개한다.
- 내용이 거의 같은 페이지는 하나의 동적 계산기와 표로 통합한다.

## 최종 의사결정

AgeCalc는 페이지 수를 빠르게 늘리는 전략보다, **사용자가 자신의 나이를 계산한 뒤 다음 생활 결정을 이해하도록 돕는 플랫폼**으로 전환해야 한다. 단기적으로는 기존 URL의 깊이와 신뢰를 높여 AdSense 승인 가능성을 개선하고, 중기적으로는 가족·교육·기념일·은퇴·건강 클러스터를 확장한다. AI는 계산 자체가 아니라 설명, 개인화, 운영 검수에 먼저 사용한다.

가장 중요한 운영 원칙은 다음 한 문장이다.

> 새 URL은 검색 키워드를 하나 더 차지하기 위해 만드는 것이 아니라, 기존 페이지가 해결하지 못하는 사용자 과제를 독립적으로 해결할 때만 만든다.
